# Stowage P1 Resource Baseline (D-126)

This file is generated by the profile rig (`make profile` /
`go test -tags=profile -run TestProfile ./internal/bench/profile/ -profile.write-baseline`).
It records the goroutine-stability and memory-footprint baselines from the
P1 profiling phase across the full driver/store matrix. Re-run `make profile`
(with -profile.write-baseline) after any change to the pipeline, lifecycle
sweeps, boot sequence, or store/vindex drivers to update these numbers.

**The goroutine-stability deltas (gIdle-g0, s2-g0) are environment-independent.**
Absolute footprint/MiB numbers are local-machine-specific and listed here for
orientation only — they will vary across machines and Go runtime versions.

---

## Rig Description

The rig lives in `internal/bench/profile/` (build tag `profile`). Run it with:

```bash
make profile
# or with full options:
STOWAGE_TEST_PG_DSN='postgres://...' \
CGO_ENABLED=1 go test -tags=profile -v -run TestProfile ./internal/bench/profile/ \
  -profile.write-baseline \
  -profile.duration 5s \
  -profile.idle 3s \
  -profile.settle 2s \
  -profile.out /tmp/stowage-pprof
```

**What the rig does:**

1. **TestProfileMatrix** — iterates the {vindex: hnsw, brute} × {store: sqlite,
   postgres} matrix sequentially. For each cell:
   - Boots a full embedded Stowage instance with the cell's drivers.
   - Idles for `-profile.idle`: samples goroutines + footprint before and after.
   - Drives `-profile.ingest` + `-profile.retrieve` concurrent goroutines for
     `-profile.duration` with block+mutex profiling enabled.
   - Captures CPU / heap / goroutine / block / mutex pprof artifacts.
   - Drains the stack; settles for `-profile.settle`; samples post-drain.
   - Asserts post-drain goroutine count S2 <= post-boot g0+eps (advisory gate).
   Postgres cells are skipped (not failed) when STOWAGE_TEST_PG_DSN is unset.
2. **TestProfileWriteBaseline** — collects results and writes this file.

**Flags:**

| Flag                     | Default | Description                                              |
|--------------------------|---------|----------------------------------------------------------|
| -profile.ingest          | 16      | Concurrent ingest goroutines                             |
| -profile.retrieve        | 16      | Concurrent retrieve goroutines                           |
| -profile.duration        | 5s      | Load duration                                            |
| -profile.idle            | 3s      | Idle observation window                                  |
| -profile.settle          | 2s      | Post-drain settle before final sample                    |
| -profile.eps             | 50      | Allowed goroutine growth before stability gate fires     |
| -profile.out             | ""      | pprof artifact dir; empty = t.TempDir() (ephemeral)     |
| -profile.dsn             | env     | Postgres DSN; defaults to STOWAGE_TEST_PG_DSN            |
| -profile.strict          | false   | Make gates fail the build (default: advisory log)        |
| -profile.write-baseline  | false   | (Re)write this file with measured numbers                |

**Load concurrency (this run):** `50` concurrent ingest (`-profile.ingest`) + `50` concurrent retrieve (`-profile.retrieve`) = **100** concurrent in-flight requests.

## Notes

- The **goroutine-stability gate** (S2 <= g0+eps) is the primary leak signal.
  "PASS" means all goroutines launched by the pipeline, sweeps, and lifecycle
  stages were collected after drain + settle.
- The **idle gate** (gIdle <= g0+eps) checks for goroutine growth at zero traffic.
  A leak here points to tickers or sweeps that accumulate at rest.
- Errors during load are tolerated — the mock gateway under concurrent load
  may return transient errors; the rig measures resources, not correctness.
- pprof artifacts are ephemeral by default (t.TempDir()). Set -profile.out
  to a persistent path and inspect with `go tool pprof`.
- The rig does NOT run under the default `go test ./...` (build tag `profile`
  guards it). It is a deliberate explicit gate (`make profile`), like `make slo`.
- Cells run **sequentially** to avoid cross-contamination of block/mutex
  profile rates and goroutine counts between backends.
